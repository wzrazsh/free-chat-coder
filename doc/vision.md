# free-chat-coder 愿景文档

更新日期：2026-04-25

## 1. 项目愿景

`free-chat-coder` 旨在成为一个本地运行的 AI 辅助开发工作台：通过 Chrome 扩展连接 DeepSeek Web，会同本地 Queue Server、Web Console 和会话存储，把浏览器中的 AI 对话能力转化为可管理、可追踪、可审批的开发协作流程。

项目追求的不是让 AI 直接接管本地工程，而是建立一套受控的协作边界：AI 负责理解需求、生成建议和提出结构化动作意图；本地系统负责任务调度、状态记录、权限隔离、人工确认和最终交付。

## 2. 目标用户

- 需要在本地项目中使用 DeepSeek 辅助编码、分析、整理文档的个人开发者。
- 希望保留浏览器 DeepSeek 使用方式，同时获得任务队列、会话记录和管理界面的高级用户。
- 对自动写文件、自动执行命令保持谨慎，希望所有高风险动作都经过人工审批的开发者。

## 3. 核心价值

### 3.1 把 AI 对话变成可管理的任务流

系统不只保存一次性聊天内容，而是围绕任务、会话、消息、动作意图和审批结果建立完整上下文，让用户可以追踪 AI 协作过程，并在必要时回看、复用或审计。

### 3.2 保留人工控制权

DeepSeek 可以生成建议、补丁草案或动作计划，但不能直接修改项目文件、重启服务或执行高风险本地操作。所有影响本地工作区的动作必须进入人工确认或后续的 patch review 流程。

### 3.3 降低浏览器自动化的不确定性

短期内系统通过 Chrome 扩展接管 DeepSeek Web 会话，完成输入、上传、读取和会话切换。中长期应逐步强化诊断、错误恢复和可替代 provider 能力，降低对页面 DOM 结构变化的脆弱依赖。

### 3.4 形成本地优先的开发工作台

数据、任务状态、审批记录和会话上下文优先保存在本地，用户可以通过 Web Console 查看和管理。系统应保持可安装、可诊断、可恢复，而不是依赖黑盒云端编排。

## 4. 产品边界

### 4.1 当前应坚持的边界

- DeepSeek 负责生成内容和结构化意图。
- Chrome 扩展负责浏览器页面内的交互和会话读取。
- Queue Server 负责队列、状态、存储、校验和 provider 调度。
- Web Console 负责可视化管理、审批和诊断入口。
- 本地文件修改必须经过明确的用户确认。

### 4.2 当前不作为主线的方向

- 不恢复 `/evolve` API。
- 不恢复自动进化、自动自修复、cron/autopilot 循环。
- 不允许 DeepSeek 直接写入项目文件或绕过人工审批。
- 不把旧 auto-evolve 任务作为新知识库或 provider 路由的基础。

## 5. 近期方向

1. 稳定 Chrome 扩展、Queue Server、Web Console、会话存储之间的基础闭环。
2. 完善动作审批、任务附件、文件交付和 patch review 的协议边界。
3. 补齐诊断能力，让扩展连接、Native Host、Queue Server、DeepSeek 页面状态更容易排障。
4. 建立知识库与上下文检索的最小可用设计，优先服务项目内文档和任务上下文。
5. 持续清理旧自动进化相关设计，避免历史方案重新进入主线实现。

## 6. 成功标准

- 用户可以把一个开发任务提交给系统，并清楚看到任务状态、关联会话和 AI 输出。
- 系统可以稳定复用 DeepSeek Web 会话完成文本协作、附件上传和结果读取。
- 所有本地高风险动作都有审批记录，用户能在执行前理解影响范围。
- 文档、架构和实现保持一致，新增能力能落到明确模块和明确边界中。
- 出错时系统能给出可执行的诊断信息，而不是只暴露浏览器自动化失败。
