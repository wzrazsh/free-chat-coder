# DeepSeek 最终回复与思考过程分离修改和测试计划

## 1. 问题结论

截图显示 DeepSeek 原页面已经将内容分成两类：

- 思考过程：位于“已思考（用时 x 秒）”折叠区域，内容是模型的推理、意图判断和回复策略。
- 回答结果：位于普通助手消息正文区域，内容是最终应展示给用户的回复。

当前 SOLO Coder / DeepSeek Agent Bridge 在同步 DeepSeek 页面内容时，把“已思考”区域中的文本也作为“最终回复”展示，导致右侧面板出现两个“最终回复”卡片：第一个是思考过程，第二个才是真正回答。

结论：问题出在桥接层的消息提取或消息分类逻辑。应当过滤 DeepSeek 的思考区域，只把正式回答区域写入 `final reply`；如果需要保留思考过程，也必须单独标记为 `reasoning` / `thinking`，不能进入最终回复列表。

## 2. 修改目标

1. 最终回复区域只展示用户可见回答，不包含“已思考”内容。
2. DeepSeek 页面存在思考区域时，桥接层能稳定识别并排除该区域。
3. 不影响普通无思考区域回答、长回答、多轮会话和历史会话同步。
4. 前端展示层不再把 reasoning 类型消息渲染为“最终回复”。

## 3. 修改方案

### 3.1 内容提取层

检查 Chrome 扩展 content script 中负责读取 DeepSeek 助手消息 DOM 的逻辑，重点排查：

- 是否使用了整段助手消息容器的 `innerText` / `textContent`。
- 是否把折叠的“已思考”区域和回答正文放在同一个消息文本里读取。
- 是否缺少对 DeepSeek reasoning DOM 节点、标题文本或折叠区域的过滤。

建议修改为：

1. 优先定位正式回答正文节点，而不是读取整个 assistant message 容器。
2. 对已知思考区域节点做排除，例如：
   - 包含“已思考”的区域。
   - reasoning / thinking 相关 class、role 或 data 属性。
   - 折叠详情区域中的推理文本。
3. 如果 DeepSeek DOM 结构无法稳定依赖 class，则采用两阶段策略：
   - 先 clone 消息 DOM。
   - 从 clone 中删除“已思考”标题及其关联内容区域。
   - 再读取剩余正文文本。

### 3.2 数据模型层

检查从 content script 发送到 queue server / web console 的消息结构，确认是否需要区分：

- `final`: 正式回答。
- `reasoning`: 思考过程。
- `status`: 执行中状态或过程提示。

如当前只有单一 `reply` / `finalReply` 字段，短期修复应直接过滤 reasoning 文本；中期可以增加 `messageType` 或 `segmentType` 字段，避免前端靠文本猜测类型。

### 3.3 前端展示层

检查 SOLO Coder 右侧面板渲染“最终回复”的逻辑：

1. 只有 `final` 类型消息可以进入“最终回复”卡片。
2. `reasoning` 类型消息默认不展示；如需要调试，可放到独立“思考过程”调试区。
3. 对历史数据做兼容：如果旧数据里已有误分类内容，可在展示时根据“已思考”标题或典型 reasoning 文本做一次保守过滤。

## 4. 验收标准

修复后，用 DeepSeek 发送“你好”这类会触发简短思考和最终回答的消息时：

- SOLO Coder 中只出现一个“最终回复”卡片。
- 卡片内容从“你好！很高兴见到你”一类正式回答开始。
- 不出现“用户发了一个简单的问候”“我需要给出一个友好、自然的回应”等思考文本。
- 多轮会话中每轮只保留对应的正式回答。

## 5. 测试计划

### TC-DS-FINAL-001：基础问候回复过滤

步骤：

1. 打开 DeepSeek 页面并确保 SOLO Coder 连接成功。
2. 发送用户消息：`你好`。
3. 等待 DeepSeek 完成回答。
4. 查看 SOLO Coder 右侧回复列表。

预期结果：

- 只展示一个“最终回复”。
- 内容为 DeepSeek 的正式回答。
- 不包含“已思考”区域内的推理文本。

### TC-DS-FINAL-002：无思考区域回答

步骤：

1. 发送一个可能不会展示思考区域的普通问题。
2. 等待回答完成。
3. 查看 SOLO Coder 同步结果。

预期结果：

- 正常显示最终回复。
- 不因为过滤逻辑导致回答为空或截断。

### TC-DS-FINAL-003：长回答与 Markdown 内容

步骤：

1. 发送需要列表、代码块或 Markdown 的问题。
2. 等待回答完成。
3. 检查 SOLO Coder 的最终回复内容。

预期结果：

- Markdown 结构尽量保留。
- 代码块、列表和段落没有明显丢失。
- 不包含思考过程。

### TC-DS-FINAL-004：多轮会话同步

步骤：

1. 连续发送 3 条消息。
2. 每条消息等待 DeepSeek 完成回答后再发送下一条。
3. 查看 SOLO Coder 中每轮最终回复。

预期结果：

- 每轮最多生成一个最终回复卡片。
- 各轮回复与对应用户消息匹配。
- 不出现跨轮重复、混入思考过程或空卡片。

### TC-DS-FINAL-005：历史会话恢复

步骤：

1. 打开已有 DeepSeek 历史会话。
2. 让 SOLO Coder 同步当前会话。
3. 检查历史助手消息提取结果。

预期结果：

- 历史消息中的最终回复仍可读取。
- 已展开或折叠的“已思考”内容不会进入最终回复。

### TC-DS-FINAL-006：前端展示兼容

步骤：

1. 准备包含 `final`、`reasoning`、`status` 三类消息的本地或模拟数据。
2. 打开 SOLO Coder 回复列表。
3. 检查不同类型消息的展示位置。

预期结果：

- `final` 渲染为“最终回复”。
- `reasoning` 不渲染为“最终回复”。
- `status` 不渲染为“最终回复”。

## 6. 回归风险

- DeepSeek DOM 结构可能变化，选择器过窄会导致提取失败。
- 过滤规则过强可能误删正式回答中包含“思考”字样的正常文本。
- 如果前端和后端都做过滤，可能造成重复处理或调试困难。

控制方式：

1. DOM 过滤优先基于节点结构，文本关键字只作为兜底。
2. 为过滤函数增加单元测试，覆盖“思考区域 + 正文”“仅正文”“正文包含思考字样”等样例。
3. 在开发日志中记录被过滤的 segment 类型和长度，但不要把 reasoning 内容写入最终回复。

## 7. 建议实施顺序

1. 定位 DeepSeek 消息提取函数。
2. 增加提取正式回答正文的 helper，并为 helper 写样例测试。
3. 修改消息同步逻辑，只发送或标记正式回答。
4. 修改 SOLO Coder 前端渲染条件，只让 final 类型进入最终回复列表。
5. 执行上述测试用例并记录结果。
